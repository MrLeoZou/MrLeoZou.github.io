见微知著的开发规范

发表于 2018-03-15 | | 阅读次数:
字数统计: 1,309 | 阅读时长 ≈ 5

本文将要分享一个颇具争议的话题：iOS开发规范，也是近期项目重构在这方面的一些体会和见解。在此先行声明，本文的规范都是个人见解，望君看完付诸一笑，切勿抨击，毕竟一千个读者就有一千个程序员，惹不起🙃。

官方规范

首先贴出Apple关于cocoa编码的规范，相信这个大家都不应该有争议吧。
Introduction to Coding Guidelines for Cocoa

这也是最基础最细致的规范，其他一切规范制定都应该以此为依据，不应该违背该规范精神，总感觉是在说宪法。
如果感觉太长了，也可以参考这个：编码规范 , 我们的规范也是参考了这个来制定的。

viewController的编码规范

其实以前我并不觉得viewController有什么严格的规范，那是因为自己孤陋寡闻，后面接触了多个老旧项目后，特别是多人维护的项目，体会是越发深刻：没有规矩不成方圆，乱的一塌糊涂，同一个项目风格百变。有一个想看看不下去的痛苦，比如目前需要重构的项目😆。

  1. 代码分块，使用#pragma mark，这一步尤其重要，而且需要制定好一套#pragma的顺序（这个顺序也是有讲究的，一般生命周期方法，事件，通知，代理/数据源，私有方法，getter和setter）当然这也是仅供参考，只是一旦定好，全部人都按照这个来。这样一来，整个项目的viewController脉络就清晰了，头是头尾是尾。

  2. 属性统一使用getter和setter，理由也是为了统一管理，阅读简洁。

  3. 理解控制器每个生命周期函数的意义，在合适的方法中添加合适的代码。比如viewDidLoad就做最基本的self.view的属性设置以及添加自view，不要把各种属性，控件之类的初始化或者属性设置放到这里，因为这基本是每个viewController最重要也第一眼阅读的方法，必须要力争做到简洁明了。

  4. viewWillAppear，viewDidAppear，viewWilDisAppear，viewDidDisappear这几个方法会在不同时刻可能会频繁触发，因此不要把只需要执行一次的代码添加到这些方法中，特别是耗时好性能的代码。这个问题也是特别容易忽视的，发现老项目中总是习惯把viewDidLoad中的代码放到viewWillAppear，这显然是没有理解这些系统方法的意义。

  5. 能抽取出去的代码就尽量抽取出去，比如：避免在控制器中直接创建子空间，而是通过自定义view来引入；比如网络请求，私有方法等，可以使用分类或工具类来处理，总之尽可能给viewController减压。

  6. 关于view的布局，也是需要一开始约定好，使用哪种布局方式，Frame，Autolayout还是其他，已经是否使用三方框架。比如重构项目中我们约定代码布局中统一使用Masonry，这样后期不会乱，每个人编码的模块别人都很容易上手。

这绝对是一个见仁见智的选择，我们不能讨论谁对谁错，我们只是需要针对自己的项目制订一套规范，毕竟这个问题谁都没有错😶。

开发速度

在页面UI上面来说，storyboard和nib确实能提高开发速度，毕竟提供的是可视化布局，一定程度上也减少了反复调整的过程，这点在使用autolayout时候更为显著。

性能体验

虽然很久之前Apple就推荐开发者使用storyboard和nib进行页面开发，不过时至今日对比纯代码来说，性能方面还是有所折扣的，尤其在布局复杂，性能吃紧的页面强烈建议使用纯代码。

团队协作

团队协作，这个主要是代码管理冲突的问题，个人观点是：不管是纯代码还是nib，前提都是需要给成员划分尽可能不冲突的模块，基础服务层最好由一人负责或者做成私有库的形式，这样才是减少代码冲突最重要的前提。另外，有一些全局hook的操作，可能对nib创建的对象属性不好处理等。
