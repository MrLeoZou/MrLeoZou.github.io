WKWebView的填坑之路

发表于 2018-03-10 | | 阅读次数:
字数统计: 2,426 | 阅读时长 ≈ 10

最近公司项目重构，webview也算是一个大头，早在WWDC 2014 就推出了WKWebView，现如今iOS已经更新到11了，如果还使用UIWebView的话实在有些过分啦。然后，就这样愉快地踏上了挖坑和填坑之路了。

WKWebview的坊间传闻

  • 更少的内存占用，相当于UIWebView的1/4 ~ 3/4
  • 与Safari相同的JavaScript引擎，同时支持更多的HTML5特性
  • 高达60fps的滚动刷新以及更快的加载速度
  • 更为细致地拆分了UIWebViewDelegate中的方法
    如上，基本都是报喜不报忧，其实按照数目来算的话，坑点肯定比优势多，只不过对比起来，坑点十个也抵不上一个优势。
    毕竟h5的话最重要的还是提升体验，因此WKWebView也必然是大势所趋

WKWebView几个tips

title

可以通过kvo监听当前webview的title属性值变化，获取到当前HTML设置的页面标题，然后动态设置APP原生的navigationBar的标题

estimatedProgress

同理，可以通过kvo监听当前webview的estimatedProgress属性值变化，表示当前webview的加载进度，通常用于设置页面进度条。当然，这是一个准确值，如果想要做成微信的体验，那需要自己先跑一个假的值，到达某个值后切换成真实值。

cookie丢失

WKWebView的cookie可以算得上的一个重伤了，不过解决起来还好。

cookie的存储

一般来说，cookie都是存储在NSHTTPCookieStorage中，WKWebView也是如此（虽然坊间传闻是存在私有空间中，其实并不是）。但WKWebView的cookie最大的问题在于：在发起请求时候，不会自动带上存放在NSHTTPCookieStorage中的cookie，这个在UIWebView是不存在的问题。这样一来导致很多重要数据丢失，影响甚大，比如h5登录态同步的问题。

丢失原因

WKProcessPool，苹果文档中有对WKProcessPool的描述：A WKProcessPool object represents a pool of Web Content process.。就是说所有的webview都共享一个WKProcessPool对象，从而实现cookie数据共享，但是这个WKProcessPool对象在APP进程被kill或者重启后会被重置，导致cookie中的数据被清空了。

NSURLProtocol不可用

在WKWebView中NSURLProtocol的使用也是一个重伤，可以说是致命的，也是最耗时最难搞的问题
需求：页面需要离线缓存，在无网的时候也能正常打开和展示，因此需要拦截webview的网络请求，获取数据存储在本地供离线时使用。
问题：在 WKWebView 上直接使用 NSURLProtocol 无法拦截请求
原因：WKWebView 在独立于 app 进程之外的进程中执行网络请求，请求数据不经过主进程。这也是WKWebView能有更好的体验一重要原因，但目前可以说是一把双刃剑了。
方案：目前业内主要解决方案是参考Apple开源的webKit2源码，其中暴露的私有API：

+ [WKBrowsingContextController registerSchemeForCustomProtocol:]

项目中参考代码：

通过自定义URLProtocol继承NSURLProtocol，并注册http/https scheme后，可以在自定义的protocol代理中拦截到网络请求。

ATS支持不足

自定义protocol后，发现项目中配置了ATS的开关，Allow Arbitrary Loads = NO，同时Allow Arbitrary Loads in Web Content = YES；此时WKWebView发起的所有http网络请求都会被阻塞。

ajax发起的POST请求丢失body

原因：由于WKWebView的网络请求是在独立线程中进行，最终会通过IPC把请求Message发送给APP线程，其中在线程间通信的时候会对请求encode成Message，出于性能的考虑，encode过程会把HTTPBody和HTTPBodyStream两个字段丢弃。
参考1：webKit2.0源码
参考2：bug repo

解决方案：

通过注入js代码，hook网页的ajax请求，重写XMLHttpRequest的send等方法，获取到原始的post body数据,然后通过原生发起该post请求获取数据后，返回给原ajax请求，模拟ajax请求完成过程。

具体代码就不贴了，涉及到公司隐私。这里提供一个来源及原理剖析就好了，相信更有意义。

如何hook ajax请求

提别提醒一下：hook的代码最好是通过重写方法，完成相关逻辑后，再调用原始的方法，比如send，不建议采取直接重写XMLHttpRequest整个对象。因为我们只需要重写几个方法，如果替换整个对象，那可能会产生一些莫名的bug，毕竟对象里面还有很多方法我们没有实现，同时也破坏了jQuery原来的方法，风险太大。

XMLHttpRequest = hookAjax;//不建议这么做
至此，我们可以顺利拦截webview发出的请求，获取到数据，并解决了post请求丢失body的问题，基本是可以支撑各种缓存需求了。

离线缓存的其他方案

manifest

manifest 属性规定文档的缓存 manifest 的位置。
HTML5 引入了应用程序缓存，这意味着 Web 应用程序可以被缓存，然后在无互联网连接的时候进行访问。

  1. 离线浏览 - 用户可以在离线时使用应用程序
  2. 快速 - 缓存的资源可以更快地加载
  3. 减少服务器加载 - 浏览器只从服务器上下载已更新/已更改的资源
    这是来自官方描述：manifest定义
    后续我们也准备换成这种方案来处理，这样一来缓存所有逻辑都可以交给h5处理了，想想真是美滋滋。
